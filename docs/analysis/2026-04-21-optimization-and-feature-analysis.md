# 软著编写 Agent 优化意见与新功能分析

## 1. 当前项目概况

软著编写 Agent 当前已经具备完整的六阶段流水线：标题分析、目录生成、说明书撰写、源码素材生成、文本风格调整、输出整理。项目同时提供 CLI 和 Streamlit UI 两种入口，并支持 OpenAI-compatible / Grok / fallback 三类生成模式。

当前实现的主要优点：

- 核心流程边界清晰，`SoftCopyrightAgent` 负责阶段编排，各阶段组件相对独立。
- 离线 fallback 能保证无 API Key 时仍可测试和演示。
- 输出物包含 Markdown、DOCX、源码目录、报告、metadata，具备材料闭环。
- 已有单元测试覆盖主要组件、CLI 低阈值端到端、LLM mock、UI 文件工具。
- 最近已补齐路径逃逸防护、prompt 打包规则、质量指标回传、远程图表默认关闭等关键风险点。

当前短板主要集中在生成质量、可配置能力、审查工作流、包发布质量、UI 交互稳定性和长期维护能力。

## 2. 优化建议

### 2.1 P0/P1 稳定性优化

#### 2.1.1 增加输出目录写入前的完整预检

现状：

- 输出阶段会在生成完成后写入多个文件。
- 路径安全已加强，但容量、权限、目录冲突、已有文件覆盖策略仍较弱。

建议：

- 在运行开始前执行输出目录预检：
  - 是否可创建目录。
  - 是否可写入临时探针文件。
  - 是否存在同名输出文件。
  - 是否启用覆盖、追加版本号或中止。
- metadata 中记录本次输出策略。

收益：

- 避免长时间生成后才因写入失败丢失结果。
- UI 体验更稳定，错误更可解释。

#### 2.1.2 引入运行级错误模型

现状：

- 多处直接抛出 `ValueError`、`LLMError`、`RuntimeError`。
- UI 中统一 `st.exception` 展示，CLI 中直接异常退出。

建议：

- 增加统一错误类型，例如：
  - `ConfigurationError`
  - `GenerationError`
  - `OutputWriteError`
  - `ReviewDraftError`
- `RunResult` 或失败报告中写入阶段、原因、是否可重试、建议修复动作。

收益：

- CLI/UI 可以给出更清楚的错误提示。
- 后续做批量任务时可以区分“配置错误”和“单个标题生成失败”。

#### 2.1.3 加强 LLM 输出校验与自动修复

现状：

- LLM JSON 解析失败后回退 fallback。
- 对 JSON schema、文件数量、章节完整性、模块覆盖的校验较基础。

建议：

- 为 analysis、outline、code generation 分别建立轻量 schema 校验。
- JSON 解析失败时先进行一次“修复 JSON”重试，而不是立即回退。
- 校验失败时把失败原因写入生成报告。

收益：

- 提升 LLM 模式成功率。
- 避免 LLM 输出轻微格式错误导致整体退化为 fallback。

### 2.2 生成质量优化

#### 2.2.1 让源码素材更接近真实项目结构

现状：

- fallback 代码生成偏模板化。
- 行数达标能力已改善，但业务逻辑仍偏浅。

建议：

- 针对不同领域生成不同架构模板：
  - AI/知识库类：embedding、retrieval、ranking、feedback、knowledge ingest。
  - 管理系统类：workflow、approval、report、role permission。
  - IoT/监控类：device、metric、alarm、rule engine。
  - 电商/供应链类：order、inventory、settlement、shipment。
- 每个模块生成模型、服务、API、测试四类文件。
- 生成代码中加入更真实的异常分支、分页、权限、审计、配置读取。

收益：

- 源码素材更符合软著审查中“功能模块与说明书一致”的要求。
- 降低所有项目输出高度相似的问题。

#### 2.2.2 文档章节增加图表和结构化说明

现状：

- 说明书主要由段落和表格组成。
- DOCX 已支持 Mermaid 回退和可选远程渲染，但文档生成阶段未系统化生成图表。

建议：

- 在章节中固定生成：
  - 系统架构图。
  - 核心业务流程图。
  - 模块调用关系图。
  - 数据流向图。
  - 部署架构图。
- 在 fallback 模式下生成 Mermaid 源码。
- UI 中通过复选框决定是否渲染为图片。

收益：

- 设计说明书更完整。
- 图表可增强审查材料的专业性。

#### 2.2.3 建立“材料一致性检查器”

现状：

- 质量评估主要关注字数、行数、模块路径覆盖。

建议：

- 新增一致性检查：
  - 说明书中的模块是否全部出现在源码目录。
  - 源码中的接口是否在说明书接口章节出现。
  - 数据表命名是否和模块 slug 对齐。
  - metadata、报告、文档标题是否一致。
- 输出一个 `consistency_report.md`。

收益：

- 比单纯字数/行数更接近真实材料质量。
- 可以在 UI 中明确告诉用户哪些地方需要人工修订。

### 2.3 UI/交互优化

#### 2.3.1 分阶段运行界面

现状：

- UI 具备阶段一审查和全卷生成能力，但流程仍偏线性。

建议：

- 明确拆成 4 个 Tab：
  - 配置与标题。
  - 规划审查。
  - 内容生成。
  - 输出与修改。
- 每个阶段保存中间状态，可单独重新运行。

收益：

- 用户可以只重生成说明书或源码，不必全流程重跑。
- 更适合人工审查场景。

#### 2.3.2 增加运行历史管理

现状：

- UI 会扫描 metadata 展示历史，但缺少对历史记录的管理。

建议：

- 历史记录页展示：
  - 标题、时间、生成模式、字数、代码行数、质量评分。
  - 打开输出目录。
  - 复制配置重新生成。
  - 删除或归档历史记录。

收益：

- 用户能管理多个软著项目。
- 便于对比 LLM 模式和 fallback 模式输出效果。

#### 2.3.3 增加敏感操作提示

建议对以下操作增加明确 UI 提示：

- 启用远程 Mermaid/Kroki 渲染：提示图表内容会发送到外部服务。
- 输入 API Key：提示只在当前运行中使用，避免写入输出文件。
- 覆盖输出目录：提示可能替换已有材料。
- 打包下载：提示包含源码和说明书全部内容。

收益：

- 降低误操作风险。
- 对软著材料这类可能包含商业信息的内容更稳妥。

### 2.4 配置和模板体系优化

#### 2.4.1 配置文件化

现状：

- CLI 参数较多。
- UI 配置只存在 session state 中。

建议：

- 支持 `softcopyright.toml` 或 `run_config.json`：
  - 标题。
  - 输出目录。
  - 文档字数、代码行数。
  - LLM provider、model、base_url。
  - 模板主题。
  - 是否生成 DOCX。
  - 是否启用远程图表。
- CLI 增加 `--config`。
- UI 支持导入/导出配置。

收益：

- 便于复现生成结果。
- 多项目批量生成时更容易维护。

#### 2.4.2 模板主题真正产品化

现状：

- `theme` 已存在，但只有 `standard` 模板实际可用。
- UI 中已有 `game`、`algorithm`、`frontend_only` 选项，但缺少对应模板。

建议：

- 明确支持主题：
  - `standard`：通用管理/业务系统。
  - `ai_knowledge`：AI、知识库、问答、搜索。
  - `frontend_app`：前端应用、小程序、可视化平台。
  - `data_platform`：数据分析、BI、报表。
  - `iot_monitoring`：设备、告警、监控。
- 每个主题提供：
  - analysis prompt。
  - outline prompt。
  - chapter prompt。
  - code prompt。
  - fallback 模块模板。

收益：

- 输出差异化更明显。
- 用户按项目类型选择即可获得更贴合的材料。

### 2.5 测试与工程质量优化

#### 2.5.1 增加快照测试

建议：

- 对 fallback 模式输出做稳定快照：
  - outline 结构。
  - metadata 字段。
  - 源码目录结构。
  - 报告关键章节。

收益：

- 防止后续修改无意改变输出合同。

#### 2.5.2 增加路径安全专项测试

现状：

- 已覆盖 `../escape.py`。

建议继续覆盖：

- Windows 盘符路径：`C:/temp/evil.py`。
- UNC 路径：`//server/share/evil.py`。
- 混合斜杠：`safe/../../evil.py`。
- 空路径、`.`、隐藏文件名。

收益：

- 对 Windows 本地工具尤其重要。

#### 2.5.3 增加打包安装测试

建议：

- 在临时目录执行：
  - `python -m build`
  - `pip install dist/*.whl`
  - 调用 LLM mock 或 prompt render 检查模板是否在包内。

收益：

- 防止 prompt、templates、静态资源遗漏。

## 3. 可能的新功能

### 3.1 批量生成模式

功能描述：

- 用户提供一个标题列表，系统批量生成多个软著项目材料。
- 每个标题独立输出目录和报告。
- 总报告汇总成功/失败、生成模式、耗时、字数、行数。

适用场景：

- 同一公司需要批量整理多个软件著作权申请材料。
- 服务商批量交付多个项目初稿。

实现建议：

- CLI 增加 `--titles-file titles.txt`。
- UI 增加批量上传区域。
- 每个任务隔离配置和输出目录。
- 失败不中断整个批次。

优先级：高。

### 3.2 人工审查工作流增强

功能描述：

- 将当前“输出草稿并等待修改”升级为可视化审查流程。
- 支持修改前后 diff、审查状态、评论备注。

可包含：

- 大纲审查。
- 章节审查。
- 源码审查。
- 一致性检查结果处理。
- 一键重新生成单个章节或单个模块。

优先级：高。

### 3.3 Word 模板导入

功能描述：

- 用户上传已有 `.docx` 模板，系统按模板样式生成最终说明书。

适用场景：

- 不同代理机构或公司有固定封面、页眉、字体、目录格式。

实现建议：

- 先支持固定模板占位符：
  - `{title}`
  - `{version}`
  - `{date}`
  - `{body}`
- 再支持从模板继承样式。

优先级：中高。

### 3.4 软著材料包一键打包

功能描述：

- 除当前单独输出外，生成一个完整交付包：
  - 设计说明书 DOCX。
  - 设计说明书 Markdown。
  - 源码 zip。
  - 生成报告。
  - 元数据。
  - 审查清单。

可选能力：

- 自动生成目录清单。
- 自动生成交付说明。
- 自动生成“需人工确认事项”。

优先级：高。

### 3.5 项目类型向导

功能描述：

- 用户不只输入标题，还可以选择业务类型、技术栈、部署方式、模块数量、目标材料风格。

示例字段：

- 项目类型：AI 问答、管理系统、数据平台、移动应用、IoT 监控。
- 技术栈：Python、Java、Go、Vue、React、UniApp。
- 数据库：MySQL、PostgreSQL、MongoDB、Redis、Elasticsearch。
- 是否包含算法模块。
- 是否包含移动端。

收益：

- 减少标题分析误判。
- 输出更接近用户真实项目。

优先级：中高。

### 3.6 本地知识库/资料输入

功能描述：

- 用户上传项目简介、需求文档、接口文档、旧代码片段，系统从资料中抽取模块和术语。

实现建议：

- 第一阶段支持 Markdown/TXT。
- 后续支持 DOCX/PDF。
- 增加资料摘要和引用映射。

收益：

- 生成内容更贴近真实项目。
- 适合已有项目补软著材料。

优先级：中。

### 3.7 质量评分仪表盘

功能描述：

- UI 展示多维评分：
  - 字数达标。
  - 代码量达标。
  - 模块一致性。
  - 文档完整性。
  - 接口覆盖。
  - 安全说明覆盖。
  - 生成内容重复度。

收益：

- 用户知道下一步该改哪里。
- 可以作为批量生成的验收依据。

优先级：中。

### 3.8 多模型对比生成

功能描述：

- 同一个标题使用多个模型或 fallback 生成大纲，用户选择一个继续。

实现建议：

- 仅阶段一/二做多模型对比，避免成本过高。
- 展示模块列表、章节结构、预估质量。

收益：

- 提升规划阶段质量。
- 降低单模型误判风险。

优先级：中。

### 3.9 本地私有化模式

功能描述：

- 明确提供“完全离线”模式：
  - 禁用 LLM。
  - 禁用远程图表。
  - 禁用外部字体。
  - 禁用所有网络请求。

收益：

- 适合包含敏感项目材料的场景。
- 降低用户对数据外传的顾虑。

优先级：高。

## 4. 推荐路线图

### 第一阶段：稳定性和可交付性

目标：确保工具可以稳定交付软著材料初稿。

建议内容：

1. 输出目录预检和覆盖策略。
2. 错误模型统一。
3. 打包安装测试。
4. 路径安全测试补充。
5. 材料包一键打包。
6. UI 敏感操作提示。

### 第二阶段：生成质量提升

目标：让输出更像真实项目材料，而不只是模板扩写。

建议内容：

1. 材料一致性检查器。
2. 领域化 fallback 模板。
3. 图表自动生成。
4. 源码模块逻辑增强。
5. 主题模板产品化。

### 第三阶段：审查和批量工作流

目标：支持真实使用中的反复审查、批量生成和历史管理。

建议内容：

1. 分阶段运行界面。
2. 单章节/单模块重生成。
3. 批量生成模式。
4. 历史记录管理。
5. 审查 diff 和评论。

### 第四阶段：资料驱动和企业化

目标：支持已有项目资料输入和更严格的私有化要求。

建议内容：

1. 本地资料输入。
2. Word 模板导入。
3. 完全离线模式。
4. 配置文件导入/导出。
5. 多模型对比规划。

## 5. 风险与注意事项

### 5.1 数据安全风险

项目面向软著材料生成，输入标题、项目资料、图表和源码可能包含商业信息。所有远程能力都应默认关闭或明确提示，包括 LLM API、Kroki 图表渲染、外部字体等。

### 5.2 生成内容合规风险

工具应定位为“材料初稿辅助生成”，不应承诺规避检测、保证通过审查或替代人工确认。报告中建议持续保留“需人工审校”的提示。

### 5.3 模板同质化风险

如果大量项目使用同一 fallback 模板，输出内容可能高度相似。需要通过项目类型、技术栈、模块结构、领域术语、图表和代码逻辑进行差异化。

### 5.4 LLM 不确定性风险

LLM 可能输出非法 JSON、不安全路径、缺失模块或不一致内容。必须持续保留 schema 校验、安全路径校验和 fallback 机制。

## 6. 建议优先级清单

| 优先级 | 项目 | 原因 |
|---|---|---|
| P0 | 完全离线模式开关 | 数据安全底线，适合软著材料场景 |
| P0 | 输出目录预检和覆盖策略 | 避免长流程结束后写入失败 |
| P1 | 材料一致性检查器 | 直接提升材料可用性 |
| P1 | 领域化模板 | 提升生成质量和差异化 |
| P1 | 一键交付包 | 提升实际交付效率 |
| P1 | 打包安装测试 | 保证发布质量 |
| P2 | 批量生成 | 适合代理服务场景 |
| P2 | 分阶段重生成 | 减少修改成本 |
| P2 | Word 模板导入 | 适配机构格式要求 |
| P3 | 多模型对比 | 提升规划质量但成本较高 |

## 7. 结论

该项目已经完成从“脚本原型”到“可运行生成工具”的关键跨越。下一步不建议优先堆叠更多模型能力，而应先补齐稳定性、数据安全、输出一致性和交付包能力。之后再做领域化模板、批量任务和审查工作流，项目会更接近真实软著材料生产工具。
