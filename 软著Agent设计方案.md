# 软著编写 Agent 设计方案

## 一、产品定位

**核心功能**：用户输入软著标题，Agent 自动生成符合软著申请标准的设计说明书（8000-10000字）和源代码文件（≥3000行），并对说明书进行 AIGC 降重处理。

**目标用户**：需要申请软件著作权的开发者/团队，降低软著材料编写的时间成本。

---

## 二、整体架构

```
用户输入软著标题
        │
        ▼
┌─────────────────┐
│  Phase 1: 标题分析 │  ← 分析标题，提取关键词、技术领域、功能模块
└────────┬────────┘
         │
         ▼
┌──────────────────────┐
│  Phase 2: 目录生成    │  ← 生成设计说明书目录结构 + 代码模块规划
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐
│  Phase 3: 说明书编写  │  ← 按目录逐章编写，控制字数 8000-10000
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐
│  Phase 4: 代码生成    │  ← 按模块生成高相关性代码，≥3000行
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐
│  Phase 5: AIGC降重    │  ← 对说明书进行改写，降低AI检测率
└────────┬─────────────┘
         │
         ▼
┌──────────────────────┐
│  Phase 6: 整理输出    │  ← 输出说明书文件 + 源代码文件
└──────────────────────┘
```

## 三、各 Phase 详细设计

### Phase 1: 标题分析

**输入**：软著标题（如"基于深度学习的智能问答系统 V1.0"）

**处理逻辑**：
1. **关键词提取**：从标题中提取技术栈、业务领域、核心功能
   - 例："基于深度学习的智能问答系统" → 技术栈: 深度学习, NLP; 领域: 智能问答; 核心功能: 问答交互、语义理解、知识检索
2. **技术方案推断**：根据关键词推断合理的系统架构
   - 前端框架、后端框架、数据库、部署方案等
3. **功能模块拆解**：将核心功能拆解为 5-8 个子模块
   - 例：用户管理、问答引擎、知识库管理、模型训练、数据统计等

**输出**：结构化的标题分析结果（JSON 格式），包含：
```json
{
  "title": "基于深度学习的智能问答系统 V1.0",
  "tech_stack": {
    "frontend": "Vue.js + Element UI",
    "backend": "Python + Flask",
    "database": "MySQL + Elasticsearch",
    "ai_framework": "PyTorch + Transformers"
  },
  "core_modules": [
    "用户管理模块",
    "智能问答引擎模块",
    "知识库管理模块",
    "模型训练与调优模块",
    "数据统计与分析模块",
    "系统配置管理模块"
  ],
  "business_domain": "智能客服/知识问答"
}
```

### Phase 3: 说明书编写

**字数分配策略**（总计 8000-10000 字）：

| 章节 | 目标字数 | 占比 |
|------|---------|------|
| 1. 引言 | 600-800 | ~8% |
| 2. 软件总体设计 | 1500-2000 | ~18% |
| 3. 模块详细设计 | 3500-4500 | ~45% |
| 4. 数据库设计 | 800-1000 | ~10% |
| 5. 系统接口设计 | 600-800 | ~8% |
| 6. 系统安全设计 | 500-600 | ~6% |
| 7. 系统部署与运维 | 400-500 | ~5% |

**编写原则**：
- **模块详细设计是核心**，每个模块包含：概述 → 功能列表 → 流程描述 → 接口签名 → 数据结构
- 流程描述使用伪代码或文字步骤，不粘贴真实代码（代码单独在源代码文件中）
- 适当使用表格（功能清单、接口参数表、数据表结构等）增加专业度
- 包含系统架构图描述（文字描述架构图内容，便于软著审查理解）

**每章编写 prompt 策略**：
```
你是一位资深软件架构师，正在为「{title}」编写软件著作权设计说明书。

当前章节：{chapter_title}
目标字数：{target_words}
上下文信息：{title_analysis}
已写内容摘要：{previous_chapters_summary}

要求：
1. 内容专业、具体，使用该领域的技术术语
2. 包含详细的功能描述和设计决策
3. 适当使用表格、列表等结构化格式
4. 避免空洞的模板化表述，每段都要有实质内容
5. 字数控制在 {target_words} 字左右
```

### Phase 4: 代码生成

**代码生成策略**：

1. **行数分配**：按模块分配代码行数，确保总计 ≥ 3000 行
   - 应用入口与配置：~200 行
   - 数据模型层：~500 行
   - 业务逻辑层：~1000 行（核心，行数最多）
   - 接口层：~500 行
   - 工具类：~300 行
   - 前端模板：~300 行
   - 测试代码：~200 行

2. **质量原则**：
   - **高相关性**：代码逻辑必须与说明书中的模块设计一一对应
   - **无需可运行**：不需要实际运行，但代码结构、命名、逻辑必须合理
   - **专业度**：使用合理的框架 API、设计模式、异常处理
   - **完整性**：每个文件有完整的类/函数定义、注释、docstring

3. **代码扩写技巧**（保证行数）：
   - 每个函数写完整的 docstring（参数说明、返回值、异常）
   - 添加详细的行内注释
   - 使用完整的异常处理（try-except-finally）
   - 数据模型包含完整的字段定义和验证逻辑
   - 接口层包含完整的请求校验和响应格式化
   - 编写完整的 CRUD 操作

### Phase 5: AIGC 降重

这是关键环节，直接决定说明书是否通过审核。

**降重策略矩阵**：

| 策略 | 具体做法 | 效果 |
|------|---------|------|
| **句式变换** | 主动句↔被动句，长句拆短句，短句合并 | ★★★★ |
| **同义词替换** | "实现"→"达成"/"完成"，"采用"→"运用"/"借助" | ★★★ |
| **语序调整** | 前置条件后移，结果前置，因果倒装 | ★★★★ |
| **增补细节** | 添加具体的技术参数、版本号、配置值 | ★★★★ |
| **领域术语** | 用更专业的术语替代通用表述 | ★★★★ |
| **表述风格** | 注入"工程师口吻"：加入设计权衡、踩坑经验 | ★★★★★ |
| **结构重组** | 调整段落顺序，合并/拆分要点 | ★★★ |
| **删除AI味** | 去除"值得注意的是"、"总的来说"、"综上"等 AI 高频词 | ★★★★★ |

**降重 Prompt 模板**：
```
你是一位人工审核专家，现在需要对以下软件设计说明书段落进行改写，降低 AI 生成痕迹。

改写要求：
1. 保持技术内容的准确性和完整性
2. 替换所有 AI 常用套话（"值得注意的是"、"需要指出的是"、"总的来说"等）
3. 使用更口语化、工程化的表达方式
4. 适当加入设计决策的理由说明（为什么这样做）
5. 调整句子结构，避免过于工整的排比句
6. 添加一些具体的数字、版本号、配置参数增加真实感
7. 保持字数基本不变

原文：
{original_text}
```

**执行方式**：逐章节降重，而非全文一次性降重（效果更好，上下文更精准）

### Phase 6: 整理输出

**输出文件清单**：

1. **设计说明书**：`{软著名称}_设计说明书.docx`
   - 格式化排版：封面、目录、页眉页脚
   - 字体：正文宋体小四，标题黑体三号/四号
   - 页边距、行距符合软著申请要求

2. **源代码文件**：`{软著名称}_源代码/` 目录
   - 按模块组织的源代码文件
   - 包含 README.md 说明项目结构和运行方式
   - 代码行数统计报告

3. **生成报告**：`{软著名称}_生成报告.md`
   - 字数统计、代码行数统计
   - 各模块对应关系表
   - AIGC 降重说明

## 四、技术实现方案

### Agent 框架选择

推荐 **Python 脚本 + Prompt 模板** 方案，便于集成到 CodeBuddy 自动化流程中。

### 核心模块

```
softcopyright-agent/
├── agent.py                 # Agent 主流程编排
├── analyzer.py              # Phase 1: 标题分析
├── outline_generator.py     # Phase 2: 目录生成
├── doc_writer.py            # Phase 3: 说明书编写
├── code_generator.py        # Phase 4: 代码生成
├── aigc_reducer.py          # Phase 5: AIGC降重
├── output_formatter.py      # Phase 6: 整理输出
├── prompts/
│   ├── analyze.md           # 标题分析 prompt
│   ├── outline.md           # 目录生成 prompt
│   ├── write_chapter.md     # 章节编写 prompt
│   ├── generate_code.md     # 代码生成 prompt
│   └── reduce_aigc.md       # AIGC降重 prompt
├── templates/
│   ├── doc_template.docx    # 说明书 Word 模板
│   └── code_structure.json  # 代码结构模板
└── utils/
    ├── word_counter.py      # 字数统计
    ├── line_counter.py      # 代码行数统计
    └── docx_formatter.py    # Word 文档格式化
```

### 流程编排伪代码

```python
class SoftCopyrightAgent:
    def run(self, title: str):
        # Phase 1: 分析标题
        analysis = self.analyzer.analyze(title)
        
        # Phase 2: 生成目录
        outline = self.outline_generator.generate(analysis)
        # 展示目录给用户确认
        user_confirmed = self.confirm_outline(outline)
        
        # Phase 3: 编写说明书（逐章）
        doc_content = {}
        for chapter in outline.chapters:
            content = self.doc_writer.write_chapter(
                chapter=chapter,
                analysis=analysis,
                previous_summary=sum(doc_content.values())
            )
            doc_content[chapter.id] = content
        
        # Phase 4: 生成代码
        code_files = self.code_generator.generate(
            analysis=analysis,
            outline=outline,
            doc_content=doc_content
        )
        
        # Phase 5: AIGC降重
        reduced_doc = {}
        for chapter_id, content in doc_content.items():
            reduced = self.aigc_reducer.reduce(content)
            reduced_doc[chapter_id] = reduced
        
        # Phase 6: 整理输出
        self.output_formatter.format(
            title=title,
            doc_content=reduced_doc,
            code_files=code_files
        )
```

## 五、关键设计决策

### 1. 用户确认点

- **目录生成后**：展示目录，用户可调整模块增删后再继续
- **说明书完成后**：用户可审阅并提出修改意见
- 其他阶段自动执行，无需人工介入

### 2. 字数/行数保障

- 每章编写后实时统计字数，不足则自动补充
- 代码生成后统计行数，不足则对核心模块扩写
- 设置安全余量：目标 9000 字（中值），实际波动 8000-10000

### 3. 代码-说明书一致性

- 代码模块与说明书模块严格对应（共享同一份模块列表）
- 接口签名在说明书和代码中保持一致
- 数据表结构在说明书和 Model 代码中保持一致

### 4. AIGC 降重重心

- **重点降重章节**：引言、总体设计（AI 味最重的部分）
- **适度降重章节**：模块详细设计（技术细节本身就不太像 AI）
- **无需降重**：数据库表结构、接口参数表（结构化数据无 AI 味）

## 六、风险与应对

| 风险 | 应对策略 |
|------|---------|
| 说明书 AI 味过重被退回 | 多轮降重 + 人工审校建议；关键章节降重 2-3 遍 |
| 代码与说明书不对应 | 共享模块元数据，生成时交叉校验 |
| 字数/行数不达标 | 编写/生成后自动校验，不足则扩写 |
| 代码过于模板化 | 根据标题关键词注入领域特有逻辑 |
| 用户对目录不满意 | 目录阶段暂停等待确认，支持增删改 |

## 七、后续扩展方向

1. **多软著批量生成**：支持一次性输入多个标题，批量产出
2. **模板库**：积累各领域的说明书模板（Web应用、移动端、AI系统、嵌入式等）
3. **AIGC 检测集成**：接入 AIGC 检测 API，量化降重效果
4. **Word 自动排版**：生成符合版权局要求的格式化 Word 文档
5. **历史版本管理**：保存生成的软著材料，便于修改复用
